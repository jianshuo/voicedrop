## What it is

VoiceDrop: open-mic-and-record iOS app. You tap record вЖТ speak вЖТ it uploads the
audio to your own cloud space вЖТ a server miner transcribes it and writes one or

## WeChat еЕђдЉЧеПЈ publish вАФ synchronous, via the Tokyo VPS relay

App вЛѓ вЖТ **еПСеЄГеЕђдЉЧеПЈиНЙз®њ** publishes a mined article as a WeChat draft and waits for the
REAL result (`created`/`updated` or the actual `errcode`), not the old fire-and-forget.

Flow: app `POST /files/api/wechat/<articleKey>` вЖТ Function reads the article JSON +
`users/<sub>/WECHAT.json` creds вЖТ POSTs `{appid,secret,cover_media_ids,article}` to the relay
and **awaits** вЖТ writes the returned `wechatMediaId`(s) back to the article JSON and the cover
cache to `WECHAT.json` вЖТ returns `{ok,created,updated}` / 502 `{errcode,errmsg}` / 409 not configured.
Idempotent: an existing draft is updated in place (no dupe).

**Why a relay:** WeChat's API only works from the IP whitelisted on the еЕђдЉЧеПЈ вАФ the Tokyo VPS
`66.42.45.128` вАФ and Cloudflare `fetch` can't route through a proxy. So a tiny always-on service
runs ON that VPS and calls `api.weixin.qq.com` directly.

- **Relay** = `wechat-relay` systemd unit on the VPS: `python3.12 /opt/wechat-relay/relay_server.py`
  bound to `127.0.0.1:8848`. **Self-contained, stdlib-only** вАФ `relay_server.py` holds the entire
  WeChat + cover toolchain (`wechat_access_token` / `md_to_wechat_html` / the `assets/wechat-covers`
  picker / `create|update|sync_wechat_drafts`); no R2 / `FILES_TOKEN`, no ASR, no Claude. **Dumb**:
  gets appid/secret + article per request, returns results; the Function persists. Code =
  `mining/relay_server.py` (single file). Deploy = `mining/deploy_relay.sh`

  restart + health). First-time provision = `mining/vps/provision.sh` (+ `wechat-relay.service`, `README.md`).
