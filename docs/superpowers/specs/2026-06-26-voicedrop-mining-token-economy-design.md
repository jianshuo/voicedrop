# VoiceDrop 挖矿 —— Token 激励经济设计

> 状态：设计稿（待评审）
> 日期：2026-06-26
> 一句话：用"区块链挖矿"叙事推广 VoiceDrop —— 说话产出内容，内容产出算力 token，token 又拿去挖更多内容。奖励是真·Claude token（产品燃料），不可提现、只在 App 内消耗，对发起方真实成本铁顶可控。

---

## 1. 目标与定位

- **要推广的核心行为**：录音 → 出文章（VoiceDrop 的真价值），以及**自传播**（老带新）。
- **手段**：把"区块链 / 挖矿 / 早鸟稀释"的叙事，套在一个**自包含的内部 token 经济**上。
- **不是**：发现金、不可提现、不涉及链上、不转售 API 接入。"区块链"仅作营销叙事，底层就是一个数据库账本。

---

## 2. 核心叙事

> **"VoiceDrop 挖矿：说话产出内容，内容产出算力。越早挖，越值钱。"**

- 奖励单位 = **Claude Code Token**（真实算力 token），不是抽象积分、不是现金。
- 天然大数字（动辄几千万/几亿）= 免费的心理爽感。
- 唯一出路是"用"——在 App 内烧掉去挖更多文章。

---

## 3. 经济引擎（闭环）

```
说话 → 出文章 → 进 VD 社区
                  │
   别人点击你的文章 ──► 读者 +1 金币，作者 +1 金币   ← 双向铸币：双方都肯点 = 病毒
                  │
   每天固定大饼 P（= 1000 元等值的真实 token）
   当天全网铸出 T 枚金币 → 每枚单价 = P / T
   夜里结算：每人到手 = 自己金币数 × (P/T)  的 token
                  │
   token 拿去挖文章（挖文章 = 花 token）────────────► 回到顶部
```

### 两套独立的反馈信号（刻意分离）

| 概念 | 作用 | 是否碰钱 |
|---|---|---|
| **点赞** | 质量 / 推荐信号，驱动社区排序与曝光 | **否**（绝不碰钱） |
| **点击（喂币）** | 价值分配：一次点击给读者和作者各 +1 金币 | **是**（铸币 → 分大饼） |

**为什么分离**：点赞一旦能换钱就会被刷、就不再是诚实的质量信号（Goodhart 定律）。把钱走另一条管子，点赞回归纯净。**明确决定：不对铸币做点赞质量加权。**

---

## 4. 关键参数（本轮拍板）

| 参数 | 值 | 说明 |
|---|---|---|
| **每日大饼真实成本上限** | **1000 元/天** | 名义 token 数巨大，但真实 API 成本铁顶 1000 元/天 |
| **新人启动额度** | **够挖 30 篇文章** | 注册即送，开局零摩擦，无需付费即可体验 |
| **金币 / token 过期** | **6 个月** | use-or-lose，防无限沉淀 + 制造"快用掉"的活跃压力 |
| **点赞质量加权** | **不做** | 保护点赞信号纯度 |

---

## 5. 成本与安全：空头支票模型（双端封顶）

这是整个方案的金融内核。

- **面值（感知价值）巨大且免费**：大饼是名义面值，写多大都不额外花钱。
- **真实成本极小**：只发生在 token 被真正烧去调 API 的那一刻——而那部分正是"内容产出"，本来就要花、也正是想要的。
- **沉淀率（breakage）**：绝大多数金币永不兑现 → 真实成本 ≪ 名义面值。

### ⚠️ 真实成本必须**同时卡在两端**，否则会被挤兑爆仓

1. **发行端（大饼）**：每天空投的 token 总量 = "1000 元能买到的真实 token 量"（按当时 Claude 价格折算，约几千万 token —— 数字本身够大够唬人，且真实成本精确 = 1000 元）。
2. **兑现端（挖矿）**：全网每天真实挖矿消耗也限速在 ≈1000 元额度内。

**为什么两端都要**：token 6 个月才过期，沉淀余额会越积越多（半年名义 outstanding 可达数十万元）。光卡发行挡不住某天大户集中挤兑。**卡兑现端才是空头支票真正的牙齿**：账面欠几个亿 token 是真的，但"收银台"每天只放 1000 元的量出去。挖矿排队 = 契合"挖矿"叙事，不违和。

> 结论：**面值巨大（真·几千万 token/天）、真实成本铁顶 1000 元/天、永不爆仓。**

### 合规

- 不可提现成人民币 → 规避资金盘 / 拉人头定性。
- 非现金奖励 → 规避 Apple 对"现金奖励绑定使用行为"的限制。
- 仍需法务/审核口径复核（见开放问题）。

## 6. 增长杠杆

1. **早鸟稀释**：人越多、互动越多 → T 越大 → 每枚越不值钱；冷清早期每枚贵得离谱 → "越早挖越值钱"的 FOMO。
2. **双向铸币**：读者 + 作者都得币 → 作者有动力拉人、读者有动力点 → 双边飞轮。
3. **互助群 = 特性而非 bug**：靠"每对用户每天只算一次"，多赚必须拉**更多真人**组成**更多新对**，且每人得**真产内容**才有东西可点。于是"互助刷量"= 每日活跃 + 持续产内容 + 拉新。先例：宠咖秀。

---

## 7. 反作弊（去重 + 账号真实性，**不碰点赞**）

| 机制 | 作用 |
|---|---|
| **每对用户每天只算一次铸币** | 把"作弊"逼成"拉真人 + 产内容"，与增长同向 |
| **每人每日点击/铸币上限** | 防单点爆量，保排行榜观感 |
| **绑 Apple 登录（账号真实性）** | 抬高"一人开百号自己跟自己组对"的成本——唯一的真漏洞 |

> 真人互助群：**敞开欢迎**。
> 单人海量假号：靠账号真实性闸门压制；即便漏几个，刷的也是"死支票"（不兑换=不花钱），故仅影响公平观感，不动钱包 → 防御紧迫度低。

## 8. 与现有架构的接入点

基于 `STATE.md` 现状：

- **账本**：每用户一个 token 余额（含批次/过期时间）。落点：R2 `users/<sub>/`（如 `wallet.json`：余额、按批次的过期、当日铸币计数、当日点击对集合）。需强一致写（HEAD 校验已有先例）。
- **铸币（点击）**：VD 社区 `CommunityPostView` 增加"点击/喂币"动作 → 写读者 & 作者各 +1，带"(reader,author,date)"去重。点赞维持独立、不碰币。
- **挖矿计费**：现挖文章免费（Miner DO / `mine.py`）。改为**消耗 token**：挖前校验余额 + 扣费 + 全网当日兑现额度（≈1000 元）限速；不足则排队/提示。新人注册发放 30 篇额度。
- **每日结算**：定时任务（cron / DO alarm，已有 6h cron 先例）：统计当日 T → 单价 P/T → 把当日金币折算成 token 入各账户（带 6 个月过期批次）。
- **展示**：钱包余额、今日单价、即将过期提醒、（可选）排行榜。
- **测试**：遵循 `agent/test/` 规则，覆盖账本、铸币去重、结算、限速、过期。

## 9. 开放问题（评审时定）

1. **名义单位换算**：金币 ↔ token 的展示倍率与文案（如何让数字"够大够唬人"又不失真）。
2. **读者也得币会不会冲淡"奖励作者"**：是否给作者更高权重（如作者 +2 / 读者 +1）？当前设计取 1:1，待定。
3. **兑现端限速的 UX**：全网额度耗尽时"挖矿排队" vs 每用户限速平滑，二选一或并用。
4. **挤兑/过期的边界**：6 个月批次过期的具体实现与提醒节奏。
5. **法务/审核口径**：非现金 token 的对外措辞，确保不触 Apple / 国内监管红线。
6. **大饼随汇率/模型价格漂移**：1000 元 → token 数的折算频率（每日重算？）。
7. **冷启动**：池子空、内容少时单价畸高的早期，如何避免被少数人独吞（这恰是"早鸟"卖点，但需观感把控）。

- 不做复杂策展奖励（Steemit curation）。
